1 — Arquitetura dos Microserviços

O Online Boutique é uma aplicação de e-commerce composta por 12 microserviços (11 na configuração base + 1 opcional para Google Cloud), cada um desenvolvido numa linguagem diferente e com uma responsabilidade única. Esta abordagem — conhecida como arquitetura de microserviços — permite que cada componente seja desenvolvido, testado, escalado e atualizado de forma independente, sem afetar os restantes.

A comunicação interna entre serviços é feita exclusivamente via gRPC, um protocolo de chamada remota de procedimentos (RPC) baseado em HTTP/2, que é mais eficiente do que REST para comunicação intra-serviço. O único ponto de entrada externo é o frontend, que expõe uma API HTTP normal para o browser do utilizador.

6 — Deployment para Múltiplos Utilizadores

Com Kubernetes, escalar para mais utilizadores é uma questão de aumentar o número de réplicas de um serviço. A estratégia inteligente é escalar primeiro os serviços que são bottleneck, não todos ao mesmo tempo.

EstratégiaComandoQuando usar
Escalar réplicas manualmentekubectl scale deployment frontend --replicas=3Sabes antecipadamente a carga esperada
HPA automático por CPUkubectl autoscale deployment frontend --cpu-percent=50 --min=1 --max=5Carga variável e imprevisível
Escalar serviço críticokubectl scale deployment productcatalogservice --replicas=3Quando identificas o bottleneck
Aumentar limites de recursosEditar YAML: cpu.limits: 500mPod usa CPU máxima mas não há mais pods a lançar

7 — Workload Crítico

O browseProduct tem peso 10 — é executado 10 vezes mais do que o checkout. O productcatalogservice é chamado em quase todas as operações e não tem cache — é o maior bottleneck potencial.

Bottlenecks por ordem de prioridade:
1. productcatalogservice — chamado em browse (×10), addToCart (×2), index, checkout. Sem cache. Escalar primeiro.
2. currencyservice — chamado em browse, viewCart, checkout. Faz chamada HTTP externa ao BCE em cada pedido.
3. cartservice + redis-cart — Redis é single-threaded. Com muitos users simultâneos, as operações ficam em fila.
4. checkoutservice — Chama 6 serviços em sequência. Latência total = soma das latências individuais.

10 — Resultados Experimentais (C1 / C2)

Esta secção apresenta os resultados dos cenários de teste executados localmente em Docker Desktop (Apple Silicon, Mac M-series). Os cenários seguem a metodologia de escalabilidade horizontal do Tema 2: baseline com 1 réplica por serviço (C1) e escalamento seletivo do bottleneck principal (C2).

Ambiente: Docker Desktop (Kubernetes local) · macOS Apple Silicon · Namespace: default · 3 perfis Locust (Casual/Normal/PowerUser) · Spawn rate: 2 users/s · Step: 5 users · Step duration: 60s · Cooldown: 15s · Critérios de quebra: p99 > 2000ms ou falhas > 5%

Cenário C1 — Baseline (1 réplica por serviço)

O C1 é o cenário de referência. Todos os serviços correm com 1 réplica — configuração padrão do Online Boutique. O objetivo é identificar o ponto de saturação e confirmar qual serviço colapsa primeiro. A distribuição de utilizadores simula comportamento realista: 30% Casual (wait 5–15s), 50% Normal (wait 2–6s), 20% PowerUser (wait 0.5–2s). O checkout usa catch_response para não contaminar métricas quando o add-to-cart falha.

Cenário C2 — Escalamento Seletivo (productcatalogservice ×3)

O C2 testa a hipótese H1: escalar o bottleneck principal desloca o ponto de saturação para mais utilizadores e reduz o p99. A única diferença face ao C1 é que o productcatalogservice é escalado para 3 réplicas antes do início dos testes — todos os outros serviços mantêm 1 réplica. 

# Preparação C2
kubectl scale deployment productcatalogservice --replicas=3 -n default
kubectl rollout status deployment/productcatalogservice -n default

# Executar (mesmo script do C1)
NAMESPACE=default ./c2.sh incremental

# Repor após o teste
kubectl scale deployment productcatalogservice --replicas=1 -n default

Análise dos resultados C1 vs C2:

O C2 apresenta resultados mistos — o ponto de quebra manteve-se nos 30 utilizadores mas o comportamento é diferente. O p99 a 20–25 users piorou significativamente (410ms→1000ms), sugerindo que escalar o productcatalogservice introduziu instabilidade no load balancing gRPC — consistent com a H5 (HTTP/2 connection affinity). Por outro lado, o RPS máximo aumentou ligeiramente (+0.50) e as falhas no ponto de quebra reduziram de 47.6% para 14.8%.

Este resultado apoia parcialmente H3: com o productcatalogservice escalado, o cartservice/Redis emerge como bottleneck dominante (evidenciado pelo p99 elevado a partir de 15 users — operações Redis começam a serializar). Escalar réplicas C# não resolve porque o gargalo é o Redis single-threaded.

CM = 0.25 RPS/réplica — ganho marginal baixo. Para validar H2, é necessário correr C3 (todos os serviços ×3) e comparar CM(C3) com CM(C2) = 0.25.
Definição operacional de Custo Marginal (CM):
CM = ΔRPS / ΔRéplicas, onde ΔRPS é a variação de throughput entre C1 e C2, e ΔRéplicas é o número de réplicas adicionadas (2 réplicas do productcatalogservice). Um CM decrescente entre cenários indica rendimentos marginais decrescentes.

Proxy de custo local: CustoLocal = Σ(CPU_pod × duração) + Σ(MEM_pod × duração), medido via kubectl top pods durante os testes.
